spring-data-redis PatternTopic 버그

현상 · Symptom

Redis Pub/Sub 구독은 정상인데 채팅·알림 메시지가 리스너에 전혀 전달되지 않음. PatternTopic("chat:*") 환경에서 STOMP 목적지가 실제 방(/sub/chat/rooms/123)이 아닌 /sub/chat/rooms/ (빈 문자열)로 발행되어 구독자에게 메시지가 라우팅되지 않았습니다.

분석 과정 · Investigation
1
실제 기기에서 채팅 수신 실패 재현
로컬이 아닌 실제 환경에서 메시지가 도달하지 않는 현상 확인
2
STOMP 발행 직전 로그 분석
channel 값이 실제 채널이 아닌 패턴 문자열임을 발견
3
spring-data-redis 이슈 트래커 조사
유사 사례와 라이브러리 내부 동작 가설 수립
4
MessageListenerAdapter 위임 구조 소스코드 분석
어댑터가 PatternTopic 패턴 문자열을 channel 인자로 전달함을 확인
5
전달 경로 문제 — 소실이 아닌 어댑터의 의도적 무시
onMessage(message, pattern) 시점엔 message.getChannel()에 실채널 정상 보유. 그러나 MessageListenerAdapter가 이를 읽지 않고 2번째 인자 pattern을 deserialize해 handleMessage(body, channel)에 채워 넘김 — Spring GitHub #3338 CLOSED / 의도된 설계
원인 · Root Cause

MessageListenerAdapter가 PatternTopic 구독을 위임할 때, 콜백의 channel 인자로 실제 채널(chat:123)이 아닌 구독 패턴(chat:*)을 전달합니다. message.getChannel()에는 실제 채널이 정상적으로 담겨 있지만, MessageListenerAdapter가 이를 읽지 않고 두 번째 인자(pattern)를 deserialize해 handleMessage(body, channel)의 channel 자리에 채워 넘기는 것이 근본 원인입니다. 이는 버그가 아닌 Spring 공식 설계(GitHub #3338 CLOSED)로, 바이트코드 레벨까지 재검증해 확인했습니다.

해결 · Resolution
  • MessageListenerAdapter를 제거하고 MessageListener 인터페이스를 직접 구현
  • onMessage(Message, byte[] pattern) 시그니처를 사용
  • message.getChannel()로 실제 채널을 추출해 STOMP 목적지를 정확히 매핑
⚠ BEFORE — MessageListenerAdapter
@Bean
MessageListenerAdapter adapter(...) {
  // channel 인자 = "chat:*" (패턴!)
  // → 목적지 /sub/chat/rooms/* // ✕ 매칭 실패
}
✓ AFTER — MessageListener 직접 구현
void onMessage(Message m, byte[] pattern) {
  String ch = new String(m.getChannel());
  // ch = "chat:123" // ✓ 실제 채널
}
결과 · Result

Chat · Notification · CS Support 3개 도메인 Redis 리스너를 일괄 수정, 다중 서버 실시간 브로드캐스트가 전면 복구되었습니다. 라이브러리 의존 코드의 내부 동작을 소스 레벨에서 이해하고 우회 설계를 도출한 경험이 되었습니다.

그 외 동시성 · 마이그레이션 이슈

②

SupportRoom 동시 close → 중복 알림 이벤트

현상같은 상담방을 거의 동시에 두 번 종료 요청 시 종료 알림이 중복 발송
원인"상태 확인 → UPDATE" 사이 race condition — 두 요청 모두 OPEN을 보고 UPDATE 실행
해결조건부 UPDATE(WHERE status != 'CLOSED') 단일 원자 연산으로 변경 · 영향 row 0이면 이벤트 스킵
결과동시 close 요청 시 이벤트 중복 발행 완전 차단
③

transferOwner ↔ kickMember 동시 요청 데드락

현상방장 위임·강퇴 동시 발생 시 데드락 — 의도한 CONCURRENT_UPDATE로 잡히지 않고 그대로 전파
원인두 메서드의 flush(락) 순서가 반대라 교착 · catch가 낙관적 락 충돌 예외만 잡아 데드락 미처리
해결4개 catch의 공통 상위 타입을 ConcurrencyFailureException으로 교체 → 낙관적 락 충돌·DB 데드락 모두 동일 처리
결과동시 요청 시 데드락이 의도한 CONCURRENT_UPDATE로 정상 처리
④

@CacheEvict 커밋 전 실행 → stale 재캐싱

현상리뷰 등록 직후 점수 조회 시 갱신 전(구) 값이 반환
원인@CacheEvict가 트랜잭션 커밋 전 실행 → 그 사이 조회가 구 데이터를 재캐싱
해결@CacheEvict 제거 → @TransactionalEventListener(AFTER_COMMIT)에서 CacheManager.evict() 직접 호출
결과stale 재캐싱이 구조적으로 불가능해짐
⑤

NOT NULL ADD COLUMN 마이그레이션 DEFAULT 누락

현상기존 row가 있는 테이블에 NOT NULL 컬럼을 DEFAULT 없이 추가하는 Flyway 마이그레이션 실패
원인NOT NULL + DEFAULT 없이 추가 시 기존 row에 값이 없어 DB 제약 위반
해결3단계 분리 — nullable 추가 → 기존 row 백필(UPDATE) → NOT NULL 제약 추가
결과마이그레이션 정상 완료 · 동일 패턴 팀 전체 공유

KAN-296

동행 참여 제한 정책 전환

기존 "동시 ONGOING 동행 2개까지" 개수 제한 → 기간이 겹치지 않는 여러 동행을 미리 계획한 정상 사용자까지 차단되는 문제를 인식. 전환 Companion에 tripStartDate/tripEndDate 추가, 기간 겹침(!tripStartDate.isAfter(p.getTripEndDate()) && !tripEndDate.isBefore(p.getTripStartDate())) 시 거부로 교체하고 개수 상태를 완전 제거했습니다. 표면 요구사항("동시 참여 제한")이 실제로 막으려는 것(기간 충돌)을 한 단계 더 파고들어 재설계했습니다.

CS 도메인 설계 기준

상태 머신(WAITING → IN_PROGRESS → CLOSED) 각 전이마다 "동시에 같은 전이를 시도하면 어떻게 되는가"를 먼저 그려보고, 그 지점마다 방어 전략을 붙이는 방식으로 설계했습니다. 기능 흐름 설계와 동시성·권한 설계를 분리하지 않고 한 번에 구성한 것이 핵심입니다.

Team Leadership

팀 리더십 — 리뷰 프로세스 개선

상황
초기엔 지정 리뷰어 없이 시간 되는 팀원 2명이 먼저 리뷰. 다들 본인 도메인으로 바빠 "누가 봐주겠지" 하는 사이 어프로브 2개를 못 채운 PR이 쌓이며 전체 진도가 막힘
과제
바이브 코딩으로 작업량이 많아 리뷰 양·질은 줄일 수 없는 상황. "AI가 짠 코드라 놓치는 부분이 있을 수 있다"는 전제로 리뷰는 더 꼼꼼해야 했고, 동시에 "누가 리뷰할지 불명확해 진도가 안 나가는" 문제를 풀어야 했음
행동
  • PR마다 담당 리뷰어 미리 지정 → "누가 봐야 하는지" 모호함 제거
  • 리뷰 필요 PR은 Slack으로 즉시 알림 → 대기 시간 최소화
