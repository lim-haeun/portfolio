Performance Optimization

메시지 조회 성능 개선

Problem

채팅방의 메시지가 증가하면서 이전 메시지를 조회하는 API의 응답 속도가 점차 느려졌습니다. 실행 계획(EXPLAIN)을 분석한 결과 불필요한 레코드 탐색이 발생하고 있었습니다.

Solution

  • EXPLAIN으로 실행 계획 분석
  • chat_room_id + id 복합 인덱스 적용
  • 커서 기반 페이지네이션 적용
  • 불필요한 Full Scan 제거

Result

Before After
조회 레코드 395 rows 20 rows
응답 시간 10.4 ~ 15.1 ms 0.06 ~ 0.09 ms
조회 방식 비효율적 탐색 복합 인덱스 활용

What I Learned

단순히 기능을 구현하는 것에서 끝나는 것이 아니라, 실행 계획을 분석하고 데이터 접근 방식을 개선하는 과정이 백엔드 성능 최적화에서 중요하다는 것을 경험했습니다.

Trouble Shooting

Redis Pub/Sub PatternTopic 문제 해결

Situation

Redis Pub/Sub를 이용해 채팅 메시지를 여러 서버로 전달하는 기능을 구현했습니다. 처음에는 PatternTopic(chat:*)을 이용해 모든 채팅방을 구독하도록 설계했습니다.

Problem

MessageListenerAdapter를 사용할 경우, 실제 채널명이 아닌 Pattern(chat:*) 문자열만 전달되었습니다.

이로 인해 roomId를 추출할 수 없어 메시지를 올바른 채팅방으로 전달하지 못했습니다.

Analysis

  • Spring Data Redis 내부 동작 분석
  • PatternTopic의 callback 파라미터 확인
  • 실제 channel 정보가 전달되지 않는 원인 파악

Solution

  • MessageListenerAdapter 제거
  • MessageListener 직접 구현
  • 실제 channel(byte[])에서 roomId 추출
  • 채팅방별 메시지 정상 브로드캐스트 구현

Result

PatternTopic 환경에서도 채널명을 정상적으로 식별할 수 있게 되었으며, 멀티 서버 환경에서 안정적으로 실시간 채팅을 처리할 수 있었습니다.
